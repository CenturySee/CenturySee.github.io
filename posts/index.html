大家好，我是算法工程笔记。 周五了，聊点轻松的。最近有个开源项目叫龙虾（OpenClaw），在圈子里挺火。简单说，它就是一个跑在你自己电脑上的AI管家——你在聊天窗口说句话，它就能自动帮你操作文件、调API、搞定一系列复杂任务。 今天这篇不搞概念轰炸，就用一个"每天早9点自动生成工作日报"的真实任务，完整拆解一遍，看看一个Agent系统从接收指令到完成交付，中间到底经历了什么。 龙虾是什么？一句话说清楚 龙虾是一个开源的、可本地部署的个人AI助理平台。 它的交互入口是你熟悉的聊天软件——Telegram、飞书、钉钉都能接。但它和普通聊天机器人的本质区别在于： 它不只是回答问题，它能自己动手干活。 操作本地文件、调用外部API、管理定时任务……这些以前需要你手动完成或者写脚本才能搞定的事，现在一句话就能启动。 你可以把它部署在Mac、Windows甚至NAS上，接入Claude、GPT等不同的大模型来驱动。 为什么火？因为它真的能"跑通" 开源Agent项目那么多，为什么偏偏龙虾火了？ 个人看法：不是因为它多完美，而是因为它提供了一整套"让大模型干活"的基础设施，并且真的跑通了。 之前很多Agent项目，要么停留在demo阶段，要么只能完成单一任务。龙虾的价值在于，它把任务调度、工具调用、多Agent协作、会话管理这些零散的能力，整合成了一个可以实际运行的系统。 好，概念不多说。直接看它怎么干活的。 阶段一：用户下达指令 你在Telegram对话框里敲了一句： “帮我每天早9点自动生成前一天的工作日报” 然后，系统内部发生了这些事： flowchart TB subgraph Phase1["阶段一：用户设置定时任务"] U["👤 用户<br>(Telegram)"] -->|"帮我每天早9点生成日报"| TG["Telegram Channel"] TG --> GW1["Gateway"] --> MA["Main Agent"] MA -->|"cron 工具 action=add<br>schedule: '0 9 * * *'<br>payload: agentTurn<br>sessionTarget: isolated"| CRON["Cron Scheduler"] end 这里面有几个关键角色： Channel（通道）：你和龙虾说话的地方。Telegram只是其中一个选项，飞书、钉钉都能换。 Gateway（网关）：整个系统的中央控制器。不管你用什么聊天软件，消息到了网关这一层，格式就统一了。它同时管着会话、配置、安全等一堆事。 Main Agent（主智能体）：真正干活的大脑。它理解了你的意图后，调用了Cron工具，设置了一个定时任务。 Cron Scheduler：定时调度器，就是个闹钟——到点了就触发。 一句话：你说了一句话 → 系统理解了意图 → 设了个定时闹钟。 到这里，你的操作就结束了。剩下的，龙虾自己来。 阶段二：闹钟响了，开始干活 第二天早上9点，Cron准时触发。接下来是最精彩的部分——多Agent协作。 flowchart TB subgraph Phase2["阶段二：每天早9点Cron触发"] TICK["⏰ 09:00 触发"] --> GW2["Gateway<br>创建 Isolated Session"] GW2 --> ORCH["协调 Agent"] ORCH -->|任务拆解| PLAN["规划3个子任务"] end subgraph Phase3["阶段三：3个SubAgent并行执行"] PLAN -->|sessions_spawn| SA["SubAgent A<br>获取日历事件<br>🔧 bash → 日历API"] PLAN -->|sessions_spawn| SB["SubAgent B<br>检查GitHub通知<br>🔧 web_fetch → GitHub"] PLAN -->|sessions_spawn| SC["SubAgent C<br>总结昨日对话<br>🔧 sessions_history + memory"] end subgraph Phase4["阶段四：汇总投递"] SA & SB & SC -->|结果返回| REG["SubAgent Registry<br>跟踪完成状态"] REG --> ORCH2["协调Agent<br>整合日报"] ORCH2 --> ANN["Announce System<br>投递到 Telegram"] ANN --> RECV["👤 用户收到日报 ✅"] end 来，拆解一下每一步到底发生了什么： ...

线上服务踩坑：硅基流动 API 从好用到不可用，只隔了一周

大家好，我是算法工程笔记。 这两天，“元宝红包”的热度还没过，但我发现了一个更极客、更“程序员”的玩法——与其跟风抢红包，不如用 AI 给自己定制一个独一无二的红包封面。 今天这篇主要聊聊，如何用腾讯元宝“手搓”红包封面，以及从这个小功能里，我们能看到什么样的产品哲学。 极简操作：微信就是显卡 很多人以为玩 AI 必须得装个 App，还要各种复杂的 Prompt。但在元宝这里，腾讯做了一个极简的减法。 你甚至不需要下载元宝 App。 操作流程简单到哪怕是家里的长辈也能学会，具体参考下面的视频： “简单来说，它把‘文生图’这个 heavy 的 AI 能力，无缝塞进了一个普普通通的聊天窗口里。” 不用配置环境，不用排队申请 Key，可以说几乎是 0 门槛。 客观评价：一个合格的 MVP 虽然方便，但作为技术博主，我们还是要客观地看它的完成度。 亮点： 零门槛：依托微信生态，触达率极高。 速度快：生成的延迟可以接受，体验流畅。 定制化：彻底告别了“抢不到大牌封面”的尴尬，人人都是设计师。 槽点： 功能单一：目前生成的仅仅是一张静态图片。 缺少灵魂：真正的高级红包封面，哪能少了气泡挂件、动态视频？ 如果把微信红包封面比作一个 refined 的产品，现在的元宝生成版，更像是一个快速上线的 MVP（最小可行性产品）。能用，好玩，但还不够“骚”。 产品思考：AI 的尽头是“嵌入” 这可能是我今天最想聊的部分。 大家有没有发现，无论是现在大火的 DeepSeek，还是国外的 ChatGPT，我们都在习惯“打开一个专门的 AI App/网页”去提问。 但元宝的这个小动作，展示了另一条路：Embedded AI（嵌入式 AI）。 与现在的元宝本身相比，用户真正想要的，可能不是手机里多装一个类似豆包、千问、ChatGPT 这样的独立 App，也不是元宝目前正在测试的“派”。用户想要的是： 一个无缝嵌入到国民级 App 中的 AI 助手。 对于腾讯：AI 应该在微信的输入框里，在朋友圈的评论区里，在公众号的排版工具里。 对于微软：如果 Office Copilot 能像当年“大眼夹”一样随叫随到（但更聪明），它绝对比一个独立的 Copilot 网页版更有粘性。 对于开发者：最近 OpenClaw (aka Moltbot/ClawBot) 也就是所谓的“Agent”之所以爆火，逻辑也是一样的——它不是让你去适应 AI，而是让 AI 跑在你熟悉的 APP、IDE、命令行里帮你干活。 “单独做一个 App 来试图取代之前的国民级应用，不仅难，而且可能从一开始就本末倒置了。” ...

Agent进化论：从Tools到Skills，Claude在下什么棋？

大家好，我是算法工程笔记。 最近 Claude 推出的 Skills 功能刷屏了技术圈。很多朋友问我：这玩意儿不就是 Tool Use 吗？换个马甲我就不认识了？ 其实不然。如果说 Tools 是 Agent 的"手"，那么 Skills 更像是让 Agent 拥有了**“外挂大脑”**。今天这篇主要聊聊，为什么 Skills 是 Agent 走向复杂工程化的关键一步。 1. 什么是 Skills？ 简单来说，Skills 是一种可动态加载的"能力插件"或"知识包"。 形式上，它就是一个文件夹，里面整整齐齐地放着： 指令 (Instructions): 比如 SKILL.md，写满了私有化的 SOP（标准作业程序）。 脚本 (Scripts): Python 脚本等，用来执行确定性的操作。 资源 (Resources): 其他辅助文件。 Agent 就像一个"乐高小人"，需要什么能力，就"咔嚓"一声把对应的 Skills 模块插上去。 2. 为什么我们需要 Skills？ 在 Agent 开发中，我们苦"Context 爆炸"久矣。Skills 的出现，主要是为了解决从 Demo 到 Production 的两个核心痛点。 1. 渐进式披露：给大脑"减负" 没有 Skills 前（单体应用模式）： 我们需要把所有的业务逻辑、具体的 Prompt、几十个 Tool 的定义，全部一股脑塞进 System Prompt 里。这就好比让一个员工在入职第一天，就把公司几百页的规章制度倒背如流。结果就是： 贵：Token 消耗巨大。 笨：上下文太长，模型注意力分散，极易产生幻觉。 有了 Skills 后（微服务模式）： Agent 采用了 “渐进式披露” (Progressive Disclosure) 的策略。 ...
